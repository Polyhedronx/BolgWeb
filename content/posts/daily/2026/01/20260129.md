## 1. 学习使用 fzf

### 下载与简单配置

```bash
# 下载 fzf
git clone --depth 1 https://github.com/junegunn/fzf.git ~/.fzf
~/.fzf/install

# 刷新终端配置
source ~/.bashrc

# 检查用
which fzf
fzf --version

# 刷新 Hash 表，让系统重新定位命令
hash -r
```

### 常用功能

```bash
# 命令历史记录搜索
Ctrl + r

# 当前目录下文件模糊匹配
Ctrl + t

# 目录跳转，直接模糊找文件夹然后回车就进去了
Alt + c
```

### 有趣的组合

```bash
# 查进程并杀掉
ps -ef | fzf

# 查找并快速打开文件，fzf 的结果被当作 vim 的参数
vim $(fzf)

# 查看 git 分支
git branch | fzf
```

### 搜索的技巧

```bash
# 默认：模糊匹配
# 精确匹配 abc
fzf 'abc'

# 反向匹配 abc
fzf !abc

# 查找以 abc 开头的文件
fzf ^abc

# 查找以 abc 结尾的文件
fzf $abc

# 空格表示“且”，比如要查到包含 abc 但不是 .txt 的文件
fzf 'abc' !txt
```

## 2. 软链接的用法

### 基本用法

> 应用场景类似于：
>
> 1. 命令全局化：将某个用户目录下的程序链接到 `/usr/local/bin`，这样全系统的用户都能直接敲命令，而不需要输入路径。
> 2. 管理配置文件：比如把配置文件放在一个 `Dotfile` 里然后存到 Git 仓库，新服务器一拉取就通过软链接分发到 home 目录。
> 3. 版本切换：比如有两个版本的 Java，可以创建一个软链接 `/usr/bin/java` 指向 java-17，想换成 java-21 时，只需删掉旧链接，重新连一下新版本，而不需要修改任何系统路径。

```bash
ln -s <源文件或目录的绝对路径> <想创建的链接名/路径>
# -s 代表 symbolic，如果不加就创建硬链接

# 例子：假设在 /home/polyhedron/scripts/hello.sh 有个脚本，想在桌面也看到它
ln -s /home/polyhedron/scripts/hello.sh /home/polyhedron/Desktop/say_hi

# 删除软链接，注意：如果是删除指向文件夹的软链接，不要加斜杠，会删除文件夹内容
rm <链接名>
```

### 查看软链接

```bash
# 最简单的方法，-l 参数表示 link
ls -l

# 查看安装的某个软件的软链接，以 fzf 为例
ls -l $(which fzf)

# 想在当前目录及子目录找出所有软链接，注意是字母 l
find . -type l

# 想看更详细的信息
find . -type l -ls
```

## 3. 加入环境变量

### 普通用户

```bash
# 例子：将用户本地 bin 目录加入 PATH
vim ~/.bashrc
export PATH="$HOME/.local/bin:$PATH"
# $HOME 代表家目录,比如 /home/polyhedron。用变量代替硬编码路径更通用。
# : 是路径的分隔符
# $PATH 放在后面意味着追加而不是覆盖。同时，如果 ~/.local/bin 里有重名的命令，优先使用本地的版本，而不是系统的版本。

# 让配置立刻生效
source ~/.bashrc

# 查看是否成功更改
echo $PATH
```

### root 用户

```bash
# 优先使用软链接，链接到默认就在所有用户 PATH 的目录 /usr/local/bin
# 以 bat 为例，此处还顺便将 batcat 改为 bat
sudo ln -s /usr/bin/batcat /usr/local/bin/bat

# 或者也可以参考普通用户，但必须写死路径，不能使用变量，比如：
export PATH="/home/polyhedron/.local/bin:$PATH"
# 但这里的坏处就是，如果删除了普通用户或者更改权限，root 的工具就失效了。
```

## 4. Linux 文件系统重新了解

### FHS (Filesystem Hierachy Standard) 标准

#### 核心系统区

- `/bin` 和 `/usr/bin` ：公共的工具栏，放着所有用户都能用的命令和软件。
- `/usr/local` ：手动安装的软件一般就在这里，防止和系统自带的软件搞混。
- `/sbin` 和 `/usr/sbin` ：放着管理系统的命令，比如磁盘格式化和关机，通常只有 root 用户能顺畅使用。
- `/boot` ：启动 Linux 用的内核文件。
- `lib` 和 `lib64` ：系统运行所需要的库文件，类似 Windows 的 .dll。

#### 配置与存放区

- `/etc` ：所有系统级别的配置文件都放在这里，比如网络配置和用户密码。
- `/home` ：每个普通用户都有自己的房间，比如 `/home/polyhedron`。
- `/root` ：root 用户的房间。
- `/opt` ：第三方软件的安装区，例如不通过包管理器安装的程序。

#### 数据与挂载区

- `/mnt` ：挂载点。例如在 WSL2 中 Windows C 盘、D 盘就在这里。
- `/media` ：通常用于自动挂载 U盘或光驱。
- `/var` ：存放经常变动的数据，最重要的是 `/var/log` 系统日志和 `/var/lib/apt` 软件缓存。

#### 虚拟 / 内存文件系统

- `/tmp` 和 `/run` ：缓存区，类似草稿纸，任何用户都能写入，但是重启会清空。
- `/proc` `/sys` `/dev` ：分别是显示进程信息、硬件信息和映射硬件设备（如硬盘）。

### 为什么要设置权限系统？

1. 安全性：环境隔离，同时 root 用户能很方便地重设普通用户的环境。
2. 职责分类：各个用户职责可以不一样，root 用户的 PATH 包含所有目录，因为需要全套工具来维护系统。

> 因此：最好保持 root 用户的环境最简化、最标准。

### `~` 的不同

- 普通 `user` ：`~` 指向 `/home/username`
- `root` 用户：`~` 指向 `/root`

> 如果 root 也住在 `/home/root`，万一 `/home` 挂载的是一个网络硬盘，而网络断了，root 连家都回不去，也就没法修复网络。所以 root 的家 `/root` 通常直接放在根分区上，确保万无一失。

## 5. 软件装在哪里？

### 包管理器

例如 `apt` 安装的：

- `/usr/bin` ：程序
- `/etc` ：配置
- `/usr/share` ：资源

### 自己下载的二进制文件

例如 `fzf` 和 `bat` ：

- 建议放在 `/usr/local/bin` ：全系统
- 或者 `/home/usernama/.local/bin` ：仅自己用
