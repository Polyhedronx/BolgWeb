## 1. Rust 中拆分模块、设计 API 的最佳实践

以构建一个库 crate，提供一个简化的 HTTP 客户端接口为目标，组织目录结构：

```markdown
src/
├─ lib.rs
├─ client.rs
├─ error.rs
└─ http/
   ├─ request.rs
   └─ response.rs
```

第一步：在 `lib.rs` 中 `creat` 根。

```rust
// lib.rs

// mod：声明模块存在
mod client;
mod error;
mod http;

// 对外暴露稳定 API
// pub use：定义 crate 的公共接口
pub use client::HttpClient;
pub use error::Error;
pub use http::{Request, Response};
// 外部用户永远只看到 crate::HttpClient 等
```

第二步：顶层模块文件 `http.rs`。

```rust
// http.rs

mod request;
mod response;

// http 模块对外 API
// 这是模块的门面文件，非常关键。
pub use request::Request;
pub use response::Response;
```

第三步：子模块实现文件，比如 `http/request.rs`。

```rust
// http/request.rs

pub struct Request {
    pub url: String,
}

impl Request {
    pub fn new(url: impl Into<String>) -> Self {
        Self { url: url.into() }
    }
}
```

第四步：其它顶层模块文件，如 `client.rs`。

```rust
// client.rs

use crate::http::Request;
use crate::error::Error;

pub struct HttpClient;

impl HttpClient {
    pub fn send(&self, req: Request) -> Result<(), Error> {
        // 省略实现
        Ok(())
    }
}
```

可维护系统的核心特征之一是：

> 修改应当局部化，影响半径可预测。

在这种组织方式下：

- 改实现：只动子模块
- 改 API：只动门面文件
- 改 crate 对外接口：只动 `lib.rs`
