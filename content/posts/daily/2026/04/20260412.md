## 1. 解决 WSL2 中打开 Neovim 无法复制文字到外部 Windows 系统粘贴的问题

### 解决问题的路径

1. 检查 WSL 配置，包括 Windows 用户配置里的 `.wslconfig` 和 WSL 终端编辑发行版内的单实例配置 `/etc/wsl.conf`，这种问题不能问不包含相关文档知识的 chatbot，除非包括微软文档库的 MCP。最好还是去找微软官方文档写配置。

2. 在调试 Neovim 之前，先验证底层的交互操作能力：使用 `echo "WSL 复制粘贴测试" | clip.exe`，到 Windows 记事本尝试粘贴，如果成功粘贴代表成功。*这里我虽然成功粘贴出 WSL，但后面的中文都是乱码，猜测是 UTF-8 转 GBK 编码出错，此处暂时不处理，等到解决 Neovim 复制粘贴问题再回头处理。*

3. 配置 Neovim 的基础剪贴板功能，编辑 LazyVim 配置文件 `~/.config/nvim/lua/config/options.lua`，写入 WSL 专属剪贴板 provider 配置，直接对接 Windows 原生剪贴板工具，无需任何第三方依赖：

   ```lua
   vim.g.clipboard = {
     name = "WSL2-Clipboard",
     copy = {
       ["+"] = { "clip.exe" },
       ["*"] = { "clip.exe" },
     },
     paste = {
       ["+"] = { "powershell.exe", "-NoProfile", "-NonInteractive", "-Command", "Get-Clipboard -Raw" },
       ["*"] = { "powershell.exe", "-NoProfile", "-NonInteractive", "-Command", "Get-Clipboard -Raw" },
     },
     cache_enabled = false,
   }
   ```

4. 接下来解决中文复制粘贴的乱码问题，我们知道：Ubuntu 等发行版 LInux 系统一般会预装转换编码的工具 `iconv`，使用方法类似 `echo "测试复制粘贴" | iconv -f UTF-8 -t GBK | clip.exe`，想着如何将这个操作融入上面所说的 Neovim 配置文件中。通过询问 AI 得到以下写法：

   ```lua
   vim.g.clipboard = {
     name = "WSL2-Clipboard-UTF8-Fix",
     copy = {
       ["+"] = { "sh", "-c", "iconv -f UTF-8 -t GBK | clip.exe" },
       ["*"] = { "sh", "-c", "iconv -f UTF-8 -t GBK | clip.exe" },
     },
     paste = {
       ["+"] = { "sh", "-c", "powershell.exe -NoProfile -NonInteractive -Command \"Get-Clipboard -Raw\" | iconv -f GBK -t UTF-8" },
       ["*"] = { "sh", "-c", "powershell.exe -NoProfile -NonInteractive -Command \"Get-Clipboard -Raw\" | iconv -f GBK -t UTF-8" },
     },
     cache_enabled = false,
   }
   ```

### 中途的无效操作

- **越界配置 WSL 文件**：把 `[interop]` 配置写到 Windows 侧的 `.wslconfig` 里，违反微软官方规范，导致 WSL 启动报「未知键」错误，互操作能力完全失效。

- **安装无效工具**：在无图形 DISPLAY 的 WSL2 里安装 `xclip`，xclip 依赖 X11 图形服务，WSL2 默认无此环境，完全无法工作，属于纯无效操作。

- **错误的健康检查命令**：在 Neovim 里执行 `:checkhealth clipboard` ，Neovim 无此独立检查模块，正确命令是 `:checkhealth provider` ，属于无效操作。

- **多余的 locale 修改**：为解决中文乱码去修改 `LANG/LC_ALL`，乱码根源是 Windows 工具的编码不匹配，和 WSL locale 无关，反而导致「locale 找不到」的系统警告，属于无效操作。

- **手动改 PATH**：手动在 `.bashrc` 里添加 Windows 系统路径，正确做法是通过 `wsl.conf` 的 `appendWindowsPath=true` 自动配置，手动修改易出现路径异常，属于多余操作。

## 2. LazyVim 中默认集成的 notice.vim 插件与 WSL2 环境的兼容性问题解决方案

> **Problem**：在 Windows Terminal 中打开 Powershell 使用 WSL2 的 Ubuntu 发行版中的 Bash 打开 Neovim 时会使终端的行数减少，关闭时会又一次使行数减少。十分影响美观和操作流畅度。

为了解决这个问题，我尝试了以下方案：

1. 更改 `.bashrc` 中的**命令提示符刷新逻辑**，由于我同时集成了 Starship 用于美化提示符，所以在这里反复改了很久，最终使 `nvim --clean` 禁用所有插件打开 Neovim 的行数减少的情况彻底消失。

2. 但是在上一步之后，我完整运行 LazyVim 的 Neovim 打开和关闭时仍然会遇到终端行数减少的 Bug，两者之间的变量就是是否运行插件，于是**定位问题**到插件上——特别是用于重绘 UI 的美化插件。

3. 之后就是逐个禁用重绘用途的插件，最终发现是 notice.nvim 这个插件的问题：

   ```lua
   -- 禁用 noice.nvim（解决 WSL2 退出 nvim 行数减少的核心问题）
   return {
     "folke/noice.nvim",
     enabled = false,
   }
   ```
4. 最后给 LazyVim 加**退出尺寸重置**，强制重置终端尺寸，就算解决了：

   ```lua
   -- 退出时强制重置终端尺寸，彻底杜绝行数变小
   vim.api.nvim_create_autocmd("VimLeave", {
     group = vim.api.nvim_create_augroup("TerminalFix", { clear = true }),
     callback = function()
       os.execute("resize > /dev/null 2>&1")
     end,
   })
   ```

## 3. CSAPP：Lab 1

这里选几个我印象深刻，想了比较久的函数实现记录下来：

```c
int isLessOrEqual(int x, int y) {
  // 这是一个比较好想的情况，即 x,y 同号时，要让 x-y <= 0 只需让 x-y-1 < 0 就行
  // 通过 &1 的操作强制取最低位，简化思考
  int cond1 = ((x + ~y) >> 31) & 1;
  // 这种情况想着如何控制异号时的比较，特别是 y < 0 < x 时，如果按第一种情况就无法判断，因为当数值很大时很可能发生溢出
  // 尝试通过右移 31 位达到忽略除了符号位以外的无关信息再去思考，同样取最低位简化思考
  // 这里陷入了如何将条件 1 也包含进来的困顿局面，但突然发现并不需要，只需在最后返回值的时候使用逻辑或
  // 所以只用考虑一种情况，那就是：y < 0 < x，也就是 x 是正数并且 y 是负数，翻译一下就行
  int cond2 = (~(x >> 31) & 1) & ((y >> 31) & 1);
  // 最后返回的是：x <= y 或者 x < 0 < y 的情况 
  return cond1 | !cond2;
}
```

最后测试的时候发现报错：

```bash
ERROR: Test isLessOrEqual(-2147483648[0x80000000],-2147483647[0x80000001]) failed... ...Gives 0[0x0]. Should be 1[0x1]
```

于是火急火燎地改了半天，最后还是在网上找到了正确解答：

```c
int isLessOrEqual(int x, int y) {
  int sx = (x >> 31) & 1;
  int sy = (y >> 31) & 1;
  int signDiff = sx ^ sy;

  int diff = y + (~x + 1); // y - x
  int diffSign = (diff >> 31) & 1;

  return (signDiff & sx) | (!signDiff & !diffSign);
}
```

不得不说这样拆分变量显得很清晰明了，同号异号分类没有用高耦合的复杂逻辑而是直接摘出一个 `signDiff` 变量，在不同的情况之间使用 `&` 操作符来保证不重不漏，十分方便讨论，可以看出这个题解的作者的集合论的基础很牢固。
